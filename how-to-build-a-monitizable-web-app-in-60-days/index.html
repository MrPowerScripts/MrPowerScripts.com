(Superman pose) I made a video not too long ago about AWS Lambda functions with the Serverless framework. Showing how to create a simple HTTP endpoint.

As the video states, there‚Äôs a number of advantages to building around the compute function architecture. For one, you don‚Äôt have a server constantly running all day when in reality it only needs about 6 seconds for X to happen. X of course, can be anything. That‚Äôs your function. Like any old programming function. So what‚Äôs my function? Send a PostMoji.

After you choose your PostMoji, enter the message and address, and finally pay for the postcard it fires off all that information to an AWS Lambda function. It‚Äôs a NodeJS based function using various libraries for the services we‚Äôre relying on. The function relies on several external services that handle the entire process of creating and mailing the PostMoji, charging the customer‚Äôs credit card, and mailing a notification to the user.

  1. Lob.com - API to create a send various types of mailing. Including letters and postcards.
  2. Stripe.com - API that handles credit card transactions securely.
  3. SparkPost.com - API to send emails
  4. MailChimp.com - Mailing List manager with API

Once the Lambda function spins up the first thing we do is check to see if they want to be bothered by us through email sometimes. If the user ticked yes we send their email address to a mailing list on MailChimp so we can poke them later.

Then we take the recipient address and send it through Lob‚Äôs address validation service. This lets us know if we can send a PostMoji to that address. From there we‚Äôll create a test postcard with Lobs test API to make sure there are no issues with the postcard data entered. If there are no issues with the test postcard we finally send the real postcard - which obviously should pass by this point. When that returns successfully we attempt to charge the users credit card.

Can I just say for a second I love stripe? Basically, the way it works is they provide you with this awesome credit card form code that gives you an amazing credit card input. Does all of the validation and looks super jazzy. It sends the credit card info directly to stripe and then returns a transaction payload. Now you send this transaction payload to your backend for processing. The transaction payload doesn‚Äôt contain any personal information of the user or their credit card. So there‚Äôs no chance of accidentally leaking user credit card info in logs and what not. So cool. Anywho.

Now finally if the payment is successful we send the user a confirmation e-mail that the PostMoji is on its way. Any exceptions met along the way are attempted to be handled gracefully. Such as if the credit card charge fails for any reason we immediately cancel the postcard order. Neato! Now you know what it is, and how it works. Why is this so fundamentally different from how we‚Äôve approached projects and side projects before? We focused our stack and goals to work within our defined limitations. Here‚Äôs some of the key things that we identified in our final setup.

  • No active running costs. GitHub Pages is hosting our frontend for free. We have no hosting costs or servers to monitor. Nor do we need to worry about scalability in the future. The one con is we‚Äôre subject to GitHub pages maintenance downtimes. The AWS Lambda is not constantly running. It only spins up to perform its function when someone makes a purchase.

  • Predictable costs. Every action within our Lambda has a predictable cost. We‚Äôre still using the free tier for many of the services, but we should be able to predict costs fairly accurately with our configuration.

  • No infrastructure ownership. With GitHub handling the frontend and AWS handling the servers, we don‚Äôt have any alarms to monitor or resources to manage. We can sleep at night and worry about our day jobs when the sun is up.

  • (Almost) limitless scalability. Our Lambda relies on other APIs. So really they‚Äôre our only bottleneck. I wouldn‚Äôt be surprised if AWS was hosting the APIs for all the services in the Lambda. They do big things yo. Right now the Lambda isn‚Äôt aware of rate limits, so that would need to be considered in the future.   • Flexibility. We‚Äôve branded this site around emoji images after we scaled back on many ideas and iterations of what this project would be. The lambda and frontend were built with this type of flexibility in mind, so they can quickly adapt to support other iterations without changing much code.

The projects that we tackled before had us spending so much time focusing on what hard problems we were going to solve. We would get far enough into the process to end up tangled by the tech we were using or our own limitations. This time around we started with identifying our own problems and limitations.
